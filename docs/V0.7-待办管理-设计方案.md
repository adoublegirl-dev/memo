# Memo 待办管理 — 设计方案 V2

> 2026-07-14 · 增加预览确认 + 多选关闭 + 历史记录 + Agent 全流程管理

---

## 一、核心交互原则

| 原则 | 说明 |
|------|------|
| **创建先预览** | 调 MCP 后返回预览卡片，用户确认才入库 |
| **关闭先确认** | 多条匹配时列出选项，用户选择后才执行 |
| **批量支持** | 一次可以关闭/操作多条 |
| **历史可查** | 完成的待办保留完整历史，可重新开启 |
| **Agent 能管** | 不只是 Dashboard，Agent 中就能增删改查 |

## 二、数据模型

### 2.1 todos 表

```sql
CREATE TABLE todos (
    id              TEXT PRIMARY KEY,
    title           TEXT NOT NULL,
    description     TEXT DEFAULT '',
    priority        TEXT DEFAULT 'medium',   -- high / medium / low
    status          TEXT DEFAULT 'todo',     -- todo / doing / done / cancelled
    session_id      TEXT,                    -- 关联会话
    memory_id       TEXT,                    -- 关联记忆
    source_agent    TEXT DEFAULT '',
    due_date        TEXT,
    completed_at    TEXT,
    reopened_at     TEXT,                    -- 重新开启时间（done→todo 时记录）
    created_at      TEXT NOT NULL,
    updated_at      TEXT NOT NULL,
    FOREIGN KEY (session_id) REFERENCES sessions(id),
    FOREIGN KEY (memory_id) REFERENCES memory_units(id)
);

CREATE INDEX idx_todos_status ON todos(status);
CREATE INDEX idx_todos_due ON todos(due_date);
```

### 2.2 todo_history 表（状态变更日志）

```sql
CREATE TABLE todo_history (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    todo_id     TEXT NOT NULL,
    from_status TEXT,              -- 变更前状态（新建时为 NULL）
    to_status   TEXT NOT NULL,     -- 变更后状态
    note        TEXT DEFAULT '',   -- 备注（完成备注/取消原因）
    agent       TEXT DEFAULT '',   -- 操作来源 Agent
    created_at  TEXT NOT NULL,
    FOREIGN KEY (todo_id) REFERENCES todos(id)
);

CREATE INDEX idx_history_todo ON todo_history(todo_id);
```

### 2.3 状态流转

```
                  ┌── 新建 ──┐
                  ↓          │
todo ──→ doing ──→ done ──→ todo (重新开启)
  │        │        │
  └────────┴──→ cancelled (不可恢复)
```

- `done` ↔ `todo`：可来回切换（重新开启）
- `cancelled`：终点，不可恢复，记录在 history 中但不再操作

## 三、交互流程设计

### 3.1 创建待办（两阶段）

```
用户: "帮我记一个待办：周五前写完数据治理文档，高优先级"

Agent 内部解析出: title/priority/due_date
    ↓
Agent 调 todo_add({ title:"写完数据治理文档", priority:"high", due:"2026-07-18", confirm:false })
    ↓
返回预览 JSON（不入库）:
{
  "preview_id": "tmp_xxx",
  "title": "写完数据治理文档",
  "priority": "high",
  "due_date": "2026-07-18",
  "source_agent": "HanaAgent",
  "confirm": false
}
    ↓
Agent 渲染预览给用户:
  ┌─────────────────────────┐
  │ 📋 待办预览              │
  │ 🔴 写完数据治理文档       │
  │    优先级: 高            │
  │    截止: 2026-07-18     │
  │    来源: HanaAgent      │
  │    [确认创建] [取消]     │
  └─────────────────────────┘
    ↓
用户: "确认"
    ↓
Agent 调 todo_add({ preview_id:"tmp_xxx", confirm:true })
    ↓
正式入库，返回: ✅ 待办已创建 | ID: abc123 | 截止 07-18
```

如果用户说"取消"或超时不管，预览丢弃，不入库。

### 3.2 关闭待办（匹配 → 选择 → 确认）

**场景 A：精确匹配 1 条**
```
用户: "把数据治理那条完成了"
    ↓
Agent 调 todo_search({ keyword: "数据治理" })
    ↓
返回 1 条匹配:
  ✅ 确定关闭这条吗？
  🔴 写完数据治理文档 (截止 07-18, HanaAgent)
    ↓
用户: "确定"
    ↓
Agent 调 todo_close({ ids: ["abc123"] })
    ↓
✅ 已关闭，完成记忆已自动记录
```

**场景 B：多条匹配**
```
用户: "把江苏公信的待办都完成了"
    ↓
Agent 调 todo_search({ keyword: "江苏公信" })
    ↓
返回 3 条匹配:
  找到 3 条待办，请输入序号选择（用逗号分隔多个，输入"全部"关闭全部）：
  1. 🔴 写完数据治理文档     📅 07-18
  2. 🟡 检查需求变更         📅 07-20
  3. 🟢 整理会议纪要          📅 -
    ↓
用户: "1,2"
    ↓
Agent 调 todo_close({ ids: ["id1", "id2"] })
    ↓
✅ 已关闭 2 条待办
```

**场景 C：模糊意图**
```
用户: "之前有个关于京东方的事情"
    ↓
Agent 调 todo_search({ keyword: "京东方" })
    ↓
返回 5 条匹配（包括已完成的和未完成的）
Agent 标注前 3 条是进行中的，后 2 条已完成
    ↓
用户选择具体操作
```

### 3.3 查看待办列表（Agent 中）

```
用户: "我有哪些待办？"
    ↓
Agent 调 todo_list({ status: "todo+doing" })
    ↓
📋 待办列表 (5 条)
🔴 [高] 写完数据治理文档       📅 07-18  HanaAgent
🔴 [高] 检查江苏公信需求变更    📅 07-20  HanaAgent
🟡 [中] 整理京东方项目资料      📅 -      WorkBuddy
🟡 [中] 更新人格引擎文档        📅 07-25  WorkBuddy
🟢 [低] 清理测试数据            📅 -      HanaAgent

用户: "只看高优先级的"
    ↓
Agent 调 todo_list({ status: "todo+doing", priority: "high" })
    ↓
📋 高优待办 (2 条)
...

用户: "最近完成的也看看"
    ↓
Agent 调 todo_list({ status: "done", limit: 5 })
    ↓
✅ 最近完成 (3 条)
...
```

### 3.4 重新开启已完成的待办

```
用户: "把'数据治理文档'那条重新打开"
    ↓
Agent 调 todo_search({ keyword: "数据治理", include_done: true })
    ↓
找到 1 条已完成:
  ✅ 写完数据治理文档 (07-17 完成)
  是否重新开启？
    ↓
用户: "是"
    ↓
Agent 调 todo_reopen({ ids: ["abc123"] })
    ↓
🔄 已重新开启，状态恢复为进行中
```

## 四、MCP 工具（6 个）

### todo_add — 创建/确认待办

| 参数 | 类型 | 说明 |
|------|------|------|
| title | string | 待办标题 |
| description | string | 详细描述 |
| priority | string | high/medium/low，默认 medium |
| due_date | string | 截止日期 |
| memory_id | string | 关联记忆 ID |
| confirm | bool | **false=返回预览不入库，true=确认入库** |
| preview_id | string | 确认时传预览 ID（confirm=true 时必填） |

### todo_search — 搜索匹配

| 参数 | 类型 | 说明 |
|------|------|------|
| keyword | string | 搜索关键词 |
| status | string | 过滤状态，默认 "todo+doing"，传 "all" 包含 done |
| limit | int | 返回数量，默认 10 |
| include_done | bool | 是否包含已完成 |

### todo_list — 列表查询

| 参数 | 类型 | 说明 |
|------|------|------|
| status | string | todo / doing / done / cancelled，多个用 + 连接 |
| priority | string | 过滤优先级 |
| limit | int | 默认 20 |
| offset | int | 分页 |

### todo_close — 关闭（完成）

| 参数 | 类型 | 说明 |
|------|------|------|
| ids | array | 待办 ID 列表（支持批量） |
| note | string | 完成备注 |

### todo_reopen — 重新开启

| 参数 | 类型 | 说明 |
|------|------|------|
| ids | array | 待办 ID 列表 |

### todo_update — 编辑

| 参数 | 类型 | 说明 |
|------|------|------|
| id | string | 待办 ID |
| title | string | 新标题 |
| priority | string | 新优先级 |
| status | string | 手动改状态 |
| due_date | string | 新截止日期 |
| description | string | 新描述 |

## 五、历史记录

每次状态变更（新建/完成/取消/重开/编辑优先级），自动写 `todo_history`：

```
todo_id: abc123
  ├── 2026-07-14 10:30  [创建]       → todo       HanaAgent
  ├── 2026-07-14 10:30  [开始]       → doing      HanaAgent  
  ├── 2026-07-17 15:00  [完成]       → done       备注: "已提交文档"  HanaAgent
  ├── 2026-07-18 09:00  [重新开启]   → todo       WorkBuddy
  └── 2026-07-18 16:00  [完成]       → done       备注: "修改后重新提交"  WorkBuddy
```

Dashboard 待办详情中显示时间线。Agent 中也可以查：

```
用户: "数据治理那条待办的历史记录"
Agent 调 todo_history({ todo_id: "abc123" })
```

## 六、与记忆系统的关联

### 创建时
- 待办的 `session_id` 自动关联当前会话
- 可绑定 `memory_id`（用户说"记住这个，然后加个待办"）

### 完成时
- 自动写入一条 EVENT 类型记忆：`"完成待办: 写完数据治理文档"`
- 记忆的 `source_agent` 继承待办的来源

### 取消时
- 不写记忆，仅在 history 中记录

## 八、跨 Agent 一致体验

用户在任何 Agent 中：
- "我有哪些待办" → 同一份列表
- "把 XXX 完成了" → 同一个确认流程
- "之前完成的 XXX 重新打开" → 同一条历史记录

Agent 不需要区分"这个待办是谁创建的"——统一操作。
