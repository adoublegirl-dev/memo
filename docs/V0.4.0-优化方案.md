# Memo V0.4.0 优化方案

> 基于 V0.3.0 实践反馈，聚焦稳定性、数据质量、用户体验。

---

## 一、总体概况

V0.3.0 完成了四项核心优化的开发和部署，在实际导入 70+ 轮历史对话后，暴露了以下问题：

| 问题 | 影响 | 优先级 |
|------|------|:---:|
| 同会话相似记忆堆积 | 检索返回多条近似内容，降低检索质量 | ⭐⭐⭐ |
| 导入意外中断后无法续传 | 大数据量导入需重头开始，浪费 API 调用 | ⭐⭐⭐ |
| 异常退出后数据库锁死 | 服务无法重启，需手动清理或重启电脑 | ⭐⭐⭐ |
| 看板只有白色主题 | 偏好深色的用户无法切换 | ⭐ |
| memo_watcher 未端到端验证 | 自动同步功能是否正常工作不确定 | ⭐⭐ |

---

## 二、CON：Consolidation 记忆合并去重

### 2.1 问题

当前 `remember_conversation` 每调用一次就生成一条独立 MemoryUnit。同一会话内连续讨论同一话题时（如 HanaAgent 开发讨论跨越 4 轮对话），会产生多条高度相似但内容略有不同的记忆。CAS 的 `refine` 关系建立了关联边，但检索时仍返回多条近似结果。

**实测案例**（导入结果）：
```
会话 "HanaAgent网页版开发"
  ├── 记忆A: 设计并实现首页到工作台导航
  ├── 记忆B: 首次可行性验证通过 (CAS: refine A)
  ├── 记忆C: 移植模型配置 (CAS: refine A, B)
  └── 记忆D: 增加大模型适配 (CAS: refine A, B, C)
```
4 条记忆内容高度重叠，检索「HanaAgent开发进展」可能返回全部 4 条。

### 2.2 方案设计

**核心思路**：在 `run_lifecycle()` 的 consolidation 阶段，对同会话内存在 `refine` 关系链的记忆做合并。

#### 合并策略

```
同会话 + 存在 REFINES 关系链 → 合并为一条
  ↓
保留最新记忆的主体内容（title + summary）
  ↓
将旧记忆的关键差异信息合并到 summary_detail
  ↓
标记旧记忆 is_superseded，建立 MERGED_INTO 关系边
  ↓
迁移旧记忆的特征词、tag_mentions 到新记忆
```

#### 合并规则

| 条件 | 行为 |
|------|------|
| 同一 session_id 内的多条记忆 | 候选合并集 |
| 存在 REFINES 关系链（A←refine←B←refine←C） | 确认可合并 |
| 最新记忆的 summary 长度 < 旧链中任一记忆的 60% | 可能是不同话题，不合并 |
| 合并后 summary + summary_detail 总和 > 2000 字符 | 仅合并标题和特征词，摘要保留最新 |

#### 新增关系类型

```python
class RelationType(str, Enum):
    ...
    MERGED_INTO = "MERGED_INTO"  # 旧记忆被合并到新记忆
```

#### LLM 辅助决策（可选，降低误合并）

对于模糊边界（摘要相似度 0.6-0.8），调用一次 LLM 判断是否应该合并：
```
以下两条记忆来自同一会话，是否应该合并为一条？
记忆A: {summaryA}
记忆B: {summaryB}
输出 JSON: {"should_merge": true/false, "reason": "..."}
```

#### 触发时机

- **自动**：`run_lifecycle()` 的 consolidation 阶段，当同会话记忆 ≥ 5 条时触发
- **手动**：新增 `memo_consolidate` MCP 工具，用户可手动触发

#### 实现要点

- 新建 `memo/lifecycle/consolidation.py` 的 `merge_session_memories()` 方法
- `memo/models` 增加 `RelationType.MERGED_INTO`
- `engine.run_lifecycle()` 的 consolidation 阶段调用合并
- 数据库迁移 003：feature_relations CHECK 约束增加 MERGED_INTO

## 三、RMP：导入断点续传

### 3.1 问题

`import_sessions.py` 按顺序处理所有会话文件，每个文件内按轮次处理。如果在中途被 `Ctrl+C` 中断或异常崩溃，已处理的会话和轮次全部丢失，重新运行需要从头开始。

以 70+ 轮对话、每轮 2-4 次 LLM 调用计算，意外中断损失可达数百次 API 调用。

### 3.2 方案设计

**核心思路**：维护一个轻量进度文件，记录已完成的 (session_file, turn_index) 对。重启时自动跳过。

#### 进度文件格式

```json
{
  "last_session_file": "2026-06-29T04-30-41-065Z_019f11a4.jsonl",
  "last_turn_index": 15,
  "completed_sessions": [
    "2026-06-15T12-43-49-201Z_019e2baa.jsonl",
    "2026-06-20T08-12-33-441Z_abc12345.jsonl"
  ],
  "current_session_turns_done": 15,
  "total_written": 128
}
```

保存在 `data/import_progress.json`。

#### 命令行参数

```bash
# 从头开始（默认）
python scripts/import_sessions.py

# 从上次中断处继续
python scripts/import_sessions.py --resume

# 强制从头开始（忽略进度文件）
python scripts/import_sessions.py --restart
```

#### 处理流程

```
读取进度文件 → 跳过 completed_sessions → 定位到 last_session_file
  ↓
从 last_turn_index + 1 继续处理当前会话
  ↓
每完成一轮 → 更新进度文件（追加写入，非覆盖，防断电丢失）
  ↓
每完成一个会话 → 更新 completed_sessions 列表
```

#### 实现要点

- `scripts/import_sessions.py` 增加 `argparse` 命令行参数解析
- 进度文件使用 JSON 格式，每次写入用临时文件 + 原子重命名
- 兼容旧版（无进度文件时等同于从头开始）

## 四、DBL：数据库生命周期管理

### 4.1 问题

V0.3.0 开发期间遇到多次数据库文件锁死，最终需要重启电脑才能释放。根因：
1. Python 进程被 `taskkill` 强制终止，SQLite WAL 文件未正常关闭
2. `pythonw` 启动的进程在后台残留，占用端口和数据库锁
3. 多次 `start` → `stop` → `start` 循环积累僵尸进程

### 4.2 方案设计

#### 4.2.1 启动时 WAL 自动修复

在 `db.init()` 中增加 WAL 健康检查：

```python
# database.py 的 init() 开头
try:
    self.conn.execute("PRAGMA wal_checkpoint(TRUNCATE)")
except sqlite3.DatabaseError:
    # WAL 文件损坏，删除重建
    logger.warning("WAL 文件损坏，正在修复...")
    self._conn.close()
    for ext in ["-wal", "-shm"]:
        Path(config.db_path + ext).unlink(missing_ok=True)
    self._conn = sqlite3.connect(config.db_path, ...)
```

#### 4.2.2 stop_all.bat 强化

```batch
@echo off
chcp 65001 >nul

REM 1. 按端口杀进程
for /f "tokens=5" %%a in ('netstat -ano ^| findstr ":9120"') do (
    taskkill /f /pid %%a 2>nul
)

REM 2. 杀所有 python 进程
taskkill /f /im pythonw.exe 2>nul
taskkill /f /im python.exe 2>nul

REM 3. 等待释放
timeout /t 2 >nul

REM 4. WAL 清理
if exist "data\memo.db-wal" del /f "data\memo.db-wal"
if exist "data\memo.db-shm" del /f "data\memo.db-shm"

REM 5. 验证
netstat -ano | findstr ":9120.*LISTENING" >nul
if errorlevel 1 (
    echo Memo services stopped.
) else (
    echo WARNING: Port 9120 still in use.
)
pause
```

#### 4.2.3 start_all.bat PID 记录

启动时将 PID 写入文件，stop 时按 PID 精准杀：

```batch
REM start_all.bat
start "" /B python "%~dp0scripts\memo_dashboard.py" >nul 2>&1
echo %!ERRORLEVEL% > "%~dp0data\dashboard.pid"
```

#### 4.2.4 数据库定期备份

在 `run_lifecycle()` 的快照阶段，自动备份数据库：

```python
import shutil
backup_path = config.db_path + f".backup.{datetime.now().strftime('%Y%m%d')}"
shutil.copy2(config.db_path, backup_path)
# 保留最近 3 个备份
```

#### 实现要点

- `memo/store/database.py` 的 `init()` 方法开头加 WAL 修复
- `start_all.bat` / `stop_all.bat` 更新
- `engine.run_lifecycle()` 快照阶段加数据库备份

## 五、DKM：Dashboard 明暗切换

### 5.1 问题

Dashboard 写死白色主题，用户无法切换。

### 5.2 方案设计

**核心思路**：CSS 变量双主题 + localStorage 记忆用户选择。

#### 实现

```html
<!-- 默认跟随系统 -->
<script>
const theme = localStorage.getItem('memo-theme') || 'light';
document.documentElement.setAttribute('data-theme', theme);
</script>
```

```css
[data-theme="dark"] {
  --bg: #0d1117; --card: #161b22; --border: #30363d;
  --text: #c9d1d9; --muted: #8b949e; --accent: #58a6ff;
  --green: #3fb950; --orange: #d2991d; --red: #f85149;
  --purple: #a371f7; --cyan: #39d2c0;
}
```

页面右上角加一个 🌙/☀️ 切换按钮。

#### 实现要点

- 在 `PAGE` HTML 模板中增加主题变量和切换按钮
- localStorage 持久化用户选择
- 图谱中连线、标签颜色也跟随主题变量

## 六、WVR：memo_watcher 端到端验证

### 6.1 问题

memo_watcher 的路径已修复，但从未实际验证过能否正确捕获 HanaAgent 会话、过滤非对话内容、自动写入 Memo。

### 6.2 方案设计

#### 6.2.1 验证清单

| 验证项 | 预期行为 |
|--------|---------|
| 监控到新会话文件 | 自动切换并记录日志 |
| 跳过 toolCall/toolResult 消息 | 不写入 |
| 跳过 thinking 块 | 不写入 |
| 跳过过短消息（<30字符） | 不写入 |
| 完整 user→assistant 轮次写入 | 自动提取摘要和特征词 |
| MVG 门控过滤低价值内容 | 跳过闲聊 |
| 异常恢复（会话文件被删除后重建） | 不崩溃，继续监控 |

#### 6.2.2 测试方法

```bash
# 1. 启动 watcher
python scripts/memo_watcher.py

# 2. 在 HanaAgent 中进行对话
# 3. 检查日志输出和数据
python -c "from memo.core.engine import engine; engine.init(); s=engine.stats(); print(s)"
```

#### 6.2.3 改进点

- watcher 当前只处理「当前最新会话文件」。如果有多个会话并行（HanaAgent 多 tab），可能漏数据
- 增加 `--all-sessions` 参数，监控所有会话文件而非仅最新一个
- 增加启动时「追赶模式」：处理自上次关闭以来新增的会话内容

#### 实现要点

- 修复后的 `scripts/memo_watcher.py` 实际运行验证
- 增加 `--all-sessions` 和追赶模式
- 增加健康检查日志（每 10 分钟输出一次统计）

## 七、实现路线图

| Phase | 内容 | 工作量 | 依赖 |
|:---:|------|:---:|:---:|
| 6.1 | DBL: 数据库生命周期管理 | 小 | 无 |
| 6.2 | RMP: 导入断点续传 | 中 | 无 |
| 6.3 | CON: Consolidation 合并去重 | 中 | DBL（数据库迁移 003） |
| 6.4 | DKM: Dashboard 明暗切换 | 小 | 无 |
| 6.5 | WVR: memo_watcher 验证 | 小 | DBL |

### 建议执行顺序

```
DBL（先解决稳定性，别让数据库再锁死）
  ↓
RMP（导入体验，大导入不怕中断）
  ↓
CON（数据质量，合并去重减少检索噪音）
  ↓
DKM + WVR（锦上添花，明暗切换 + watcher 验证）
```
