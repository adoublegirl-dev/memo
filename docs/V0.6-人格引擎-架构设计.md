# Memo 人格引擎 — 架构设计文档 v2

> 2026-07-10 · V0.6~V0.8 设计稿 · 第二轮讨论确认版

---

## 一、目标

在 Memo 记忆库之上，构建一个携带用户人格意识的推理层。让 Agent 不只是"能查你说了什么"，而是"了解你是个什么样的人，能替你判断"。

---

## 二、人格维度设计（10 维）

原 6 维 + 参考 PersonaForge (ACL 2026)、Agent Persona Engine (8-layer)、BESSER User Modeling Language (9-dim)、PERSONA.md (10-layer) 后扩展为 10 维：

| # | 维度 | 含义 | 参考来源 | 示例 |
|---|------|------|---------|------|
| 1 | **value** | 核心价值观 | 原设计 | "极度重视信任与诚实" |
| 2 | **decision** | 决策模式 | 原设计 + PersonaForge | "决策前需要看数据支撑" |
| 3 | **identity** | 身份标签 | 原设计 + Agent Persona Engine | "法务AI产品经理，非技术背景" |
| 4 | **preference** | 偏好倾向 | 原设计 + BESSER | "审美：精致UI、简洁直接" |
| 5 | **sensitivity** | 敏感话题 | 原设计 | "对C盘空间紧张反复提及" |
| 6 | **relationship** | 关系强度 | 原设计 + Agethos RelationshipBook | "京东方项目优先级更高" |
| 7 | **knowledge** | 知识边界 | Agent Persona Engine "Limitations" | "不懂前端框架，不碰CSS" |
| 8 | **communication** | 沟通风格 | PERSONA.md + Agent Persona Engine | "偏好短句、代码示例、拒绝客套话" |
| 9 | **mental_model** | 思维模型 | Agent Persona Engine "Mental Models" | "倾向于第一性原理分析问题" |
| 10 | **emotion** | 情绪特征 | PersonaForge "Dynamic State" + Agethos | "对技术细节兴奋、对重复劳动厌倦" |

每一维可有多条断言。每条断言带 `confidence` 和 `evidences`（引用记忆 ID）。

### 自定义维度（留口子，后期做）

`persona_assertions` 表的 `dimension` 字段不做 CHECK 约束，允许用户通过管理端新增自定义维度。系统预设的 10 维保持不变，自定义维度标记 `is_custom=1`。

## 三、提炼策略：双模式自动切换

根据用户是否有历史数据，自动选择路径：

```
首次启动
    │
    ├── 记忆数 > 0 → 模式 B：批量建基线
    │      ① 取 L2（手动）+ L1（高价值）全部记忆
    │      ② L0 记忆按 gating_score 取前 20%
    │      ③ 合并后按 10 维逐维提炼 → 初始 persona_assertions
    │      ④ 之后切换到增量模式
    │
    └── 记忆数 = 0 → 模式 A：渐进积累
           每写入一条新记忆 → 增量检查是否影响人格
           推理式更新（印证/补充/推翻）
```

批量和增量都用 LLM 提炼，输出 JSON 断言列表，自动去重和冲突处理。

### 批量提炼采样策略

```
采样池 = L2 全部 + L1 全部 + L0 中 gating_score 前 20%
采样后按维度分组，每个维度取相关记忆最多 30 条
单次提炼的 token 上限：采样记忆总字符数 ≤ 20,000
```

## 四、用户管理端设计

### 4.1 功能清单

| 功能 | 说明 |
|------|------|
| 查看 | 按维度分组展示全部断言，标注 confidence、来源记忆、更新时间 |
| 编辑 | 修改断言内容、confidence、维度 |
| 删除 | 软删除（is_superseded=1），保留审计链 |
| 新增 | 手动添加断言，标记 signal_level=L2，confidence=1.0 |
| 锁定 | 标记 `locked=1`，后续增量提炼不再修改此断言 |
| 回看 | 点击 evidence 中的记忆 ID，弹窗查看原始对话 |

### 4.2 编辑后的状态处理

用户编辑一条断言后的级联操作：

| 操作 | 断言状态 | 关联记忆 | 后续提炼 |
|------|---------|---------|---------|
| 修改内容 | 保留原断言（superseded），新建改后断言 | 不变 | 新断言 `locked=0`，后续可被再次更新 |
| 修改 confidence | 直接更新该条 | 不变 | 下次提炼以新 confidence 为基准 |
| 删除 | is_superseded=1 | 不删除记忆 | 该维度屏蔽，不参与提炼 |
| 锁定 | locked=1 | 不变 | 提炼时跳过此断言 |

**核心原则**：编辑只影响人格断言表，不动记忆表。用户改的是"我对自己的理解"，不是"我做过的事"。

### 4.3 界面设计

新增看板页面「🧬 人格画像」：

- 左栏：维度导航（10 维 + 自定义维度）
- 中栏：断言卡片列表，每张卡显示断言内容 + confidence 进度条 + 来源记忆数
- 右栏：详情面板（编辑/锁定/删除按钮 + evidence 记忆列表）
- 顶栏：「提炼」按钮（手动触发增量提炼）

前端实现风格与看板一致（深色主题、D3 可选）。

## 五、人格路由器

### 5.1 灵敏度等级

5 级可配置，默认 0.30：

| 等级 | 阈值 | 标签 | 行为 |
|:---:|:---:|------|------|
| 1 | 0.15 | 全部依赖人格 | 几乎所有问题都先看人格 |
| 2 | 0.30 | 大部分依赖人格 | **默认**：高相关走人格，低相关混合 |
| 3 | 0.50 | 中等依赖人格 | 只有明确相关才走人格 |
| 4 | 0.65 | 少量依赖人格 | 人格仅修饰语气 |
| 5 | 0.80 | 几乎不依赖 | 仅极端匹配时触发 |

### 5.2 修改方式

- 看板「人格画像」页顶部下拉选择
- Agent 对话中说「把人格灵敏度调到2」
- 存储在 `config` 表或 `persona_settings` 中，持久化

### 5.3 路由逻辑

```
用户问题 → 向量相似度 × 人格断言
    │
    ├── score ≥ threshold → 人格通道
    │     回复 = 人格立场 + 引用依据
    │
    ├── 0.15 ≤ score < threshold → 混合通道
    │     回复 = 记忆检索结果 + 人格风格修饰
    │
    └── score < 0.15 → 经验通道
          回复 = 记忆检索 + 联网 + 标注"跟你平时不太一样"
```

### 5.4 偏离检测

人格通道内如果发现用户当前提问与历史人格矛盾（如偏好简洁却问炫酷特效），不直接否决，而是：

> 以你的风格通常会选简洁方案。不过这次你想要更丰富的效果——确认一下：是这次特殊，还是偏好有变化？

如果用户确认变化 → 触发增量提炼，生成新断言替代旧断言。

## 六、数据模型

```sql
CREATE TABLE persona_assertions (
    id              TEXT PRIMARY KEY,
    dimension       TEXT NOT NULL,        -- 10维之一 或 自定义
    assertion       TEXT NOT NULL,        -- 断言内容
    confidence      REAL DEFAULT 0.5,     -- 置信度 0~1
    evidences       TEXT,                 -- JSON: ["mem_id1","mem_id2"]
    signal_level    INTEGER DEFAULT 1,    -- 0=自动低置信 1=多次印证 2=手动/初始基线
    is_superseded   INTEGER DEFAULT 0,
    superseded_by   TEXT,
    locked          INTEGER DEFAULT 0,    -- 用户锁定标志
    is_custom       INTEGER DEFAULT 0,    -- 自定义维度标志
    created_at      TEXT NOT NULL,
    updated_at      TEXT NOT NULL,
    last_refreshed  TEXT
);

CREATE TABLE persona_settings (
    key   TEXT PRIMARY KEY,
    value TEXT NOT NULL
);
-- 初始: ('sensitivity_level', '2'), ('last_baseline_at', '')
```
