# Memo V0.3.0 优化方案

> 基于 D-Mem (arXiv 2603.18631)、Cognis (arXiv 2604.19771)、Mem0 Memory Decay、PAMU、Oblivion 等前沿研究，结合 Memo 现有架构设计。

---

## 一、总体设计原则

四条优化共同构成一个**写入→加权→检索→更新**的完整信号链：

```
对话片段 → [MVG 门控] → 写入 → [CAS 变更检测]
                  ↓                      ↓
            [SCB 会话加成] + [ESA 显式放大]
                         ↓
                  检索时综合加权
```

| 优化 | 作用阶段 | 核心思路 |
|------|---------|---------|
| MVG（记忆价值门控） | 写入前 | 低成本 LLM 预判，不重要的不写入 |
| CAS（变更感知） | 写入后 | 主动检测是否推翻旧事实，自动标记旧记忆失效 |
| SCB（会话凝聚力加成） | 写入时 | 同会话内记忆之间自动建立更强的边权重 |
| ESA（显式信号放大） | 写入+检索 | 用户主动要求记住的内容，全链路提升权重 |

---

## 二、MVG：记忆价值门控

### 2.1 问题

参考 Mem0 生产环境真实数据：**10,134 条记忆中有 97.8% 是垃圾**（[Issue #4573](https://github.com/mem0ai/mem0/issues/4573)）。当前 Memo 的 memo_watcher 自动同步会无差别写入所有对话，产生大量噪音。

### 2.2 学术参考

- **D-Mem** 的 Multi-dimensional Quality Gating：通过多维度质量门控动态决定走快速路径还是完整路径
- **Oblivion** 的 uncertainty-gated retrieval：基于不确定性决定是否访问记忆
- **GAM** 的 JIT 原则：离线阶段只保留简单但有用的记忆

### 2.3 方案设计

**核心思路**：写入前用一次轻量 LLM 调用做「是否值得长期记忆」的判断，不重要的直接跳过。

#### 评分维度（4 维，1-5 分）

| 维度 | 说明 | 示例（高分→低分） |
|------|------|------------------|
| **事实性** | 是否包含可被未来检索锚定的具体事实 | 「决定用 PostgreSQL」=5 / 「嗯好的」=1 |
| **决策性** | 是否包含决策、偏好、或结论 | 「用户偏好深色模式」=5 / 「看板地址 localhost:9120」=1 |
| **持久性** | 信息是否在 1 周后仍有参考价值 | 「技术选型方案」=5 / 「今天午饭吃啥」=1 |
| **独特性** | 是否引入了新的概念、关系、或事实 | 「ELO 算法初始分 1200」=5 / 重复问候 =1 |

#### 门控规则

```
总分 = 事实性×0.3 + 决策性×0.3 + 持久性×0.25 + 独特性×0.15

总分 ≥ 3.0 → 写入（正常流程）
总分 < 3.0 → 跳过（记录跳过原因到日志）
```

#### LLM 调用设计

单次轻量调用，用 `gpt-4o-mini` / `deepseek-v4-flash`，输出 JSON：

```python
GATING_PROMPT = """判断以下对话片段是否值得存入长期记忆。

评分标准（1-5）：
- factuality: 是否包含具体事实（不是闲聊）
- decision: 是否包含决策、偏好或结论
- persistence: 一周后是否还有参考价值
- uniqueness: 是否引入了新信息

对话：
{conversation}

输出 JSON：{"factuality": N, "decision": N, "persistence": N, "uniqueness": N}"""

# 约 200 tokens 输入，50 tokens 输出，成本极低
```

#### 降级策略

- LLM 不可用时：使用规则降级（字符数 < 30 → 跳过；纯问候/确认词 → 跳过）
- 总分为 2.5-3.0（灰色地带）：写入但标记 `confidence=low`，后续 consolidation 时可能被合并或遗忘

#### 实现位置

- `memo/extraction/gating.py`（新文件）
- `memo_watcher.py` 中调用 `remember_conversation` 前插入门控检查
- `engine.remember_conversation` 增加 `skip_gating=False` 参数（手动调用时跳过高开销门控）
- `MemoClient.remember()` 增加 `importance_check=True` 参数

## 三、SCB：会话凝聚力加成

### 3.1 问题

当前赫布边权重只看特征词的共激活次数，不区分「同一场对话中紧密讨论的」和「跨会话偶尔关联的」。同一会话内反复讨论的概念应该有更强的关联。

### 3.2 学术参考

- **Cognis** 的 Temporal Boosting：时间敏感查询获得额外权重提升
- **Mem0 Memory Decay**：最近访问的记忆获得 1.0x → 0.3x 的 soft boost
- **PAMU** 的 Sliding Window + EMA：滑动窗口捕获短期变化，指数移动平均建模长期趋势
- **Bjork 双强度模型**（Memo 已采用）：storage_strength 只增不减，retrieval_strength 衰减

### 3.3 方案设计

**核心思路**：在现有赫布权重更新公式中增加会话级加成因子。

#### 当前赫布权重更新

```python
Δw = η × (1 - w) × S_semantic × co_occurrence_boost
```

#### 改进后

```python
Δw = η × (1 - w) × S_semantic × co_occurrence_boost × session_boost

session_boost = 1.0 + α × min(session_co_occurrence_count / 5, 1.0)
# α = 0.5（会话加成系数）
# session_co_occurrence_count：这两个特征词在当前会话内已共现了多少次
```

**效果**：

| 场景 | old Δw | new Δw |
|------|--------|--------|
| 跨会话第一次共现 | 0.05 × 0.5 × 1.0 = 0.025 | 0.05 × 0.5 × 1.0 × 1.0 = 0.025（无变化） |
| 同会话第 3 次共现 | 0.05 × 0.5 × 1.2 = 0.030 | 0.05 × 0.5 × 1.2 × 1.3 = **0.039**（+30%） |
| 同会话第 6 次共现 | 0.05 × 0.5 × 1.4 = 0.035 | 0.05 × 0.5 × 1.4 × 1.5 = **0.053**（+50%） |

#### 扩散激活时也引入会话加成

```python
# 图搜索时，如果被激活的节点与当前查询来自同一 session，加成 1.2x
if mem.session_id == current_session_id:
    activation *= 1.2
```

#### 实现要点

- `FeatureRelation` 表增加 `last_session_id` 字段（记录最近一次共现所在的会话）
- `graph_store.upsert_relation()` 增加 `session_id` 参数
- 图扩散激活时，`def recall()` 已经接收 `current_session_id`（V0.2.0 已新增），在此处应用加成
- 参数可配：`session_boost_alpha = 0.5`、`session_spread_boost = 1.2`

## 四、ESA：显式信号放大

### 4.1 问题

用户主动说「记住这个」的内容，应该比 watcher 自动抓取的噪音有更高的检索权重。但目前二者没有区分。

### 4.2 学术参考

- **PAMU** 的显式偏好感知：用户明确陈述的偏好获得更高权重
- **Salesforce Agentic Memory** 的 confidence scoring：写入和读取都有置信度门控
- **Explicit vs Implicit Feedback** 理论：显式信号精确但稀疏，隐式信号丰富但嘈杂——二者加权融合

### 4.3 方案设计

**核心思路**：在记忆的整个生命周期中，根据来源打上不同的权重标签，检索时加权排序。

#### 三级信号强度

| 级别 | 来源 | initial_weight | 行为 |
|:---:|------|:---:|---|
| **L2 显式** | 用户主动说「记住」/ `memo_remember` 手动模式 | 1.5 | 检索时 ×1.5；遗忘时 storage_strength 额外 +0.02/次 |
| **L1 高价值自动** | watcher 捕获 + MVG 总分 ≥ 4.0 | 1.0 | 正常流程 |
| **L0 普通自动** | watcher 捕获 + MVG 总分 3.0-4.0 | 0.7 | 正常流程，但初始权重打折 |

#### MemoryUnit 新增字段

```python
signal_level: int = 0  # 0=L0普通, 1=L1高价值, 2=L2显式
```

#### 检索时的加权融合

```python
# RRF 融合前，对每条结果的 score 乘以信号系数
signal_multiplier = {0: 0.7, 1: 1.0, 2: 1.5}
adjusted_score = raw_score * signal_multiplier[mem.signal_level]
```

#### 遗忘保护

L2 显式记忆享有遗忘豁免：

```python
# forgetting.py 中
if mem.signal_level == 2:
    # 不参与自动遗忘衰减
    # 只能被用户手动删除或 is_update_of 覆盖
    continue
```

#### 实现要点

- `engine.remember()` 手动模式写入时，自动设置 `signal_level=2`
- `engine.remember_conversation()` 自动模式，结合 MVG 总分：
  - MVG 总分 ≥ 4.0 → `signal_level=1`
  - MVG 总分 < 4.0 → `signal_level=0`
- `memo_watcher.py` 的自动写入默认走 MVG 判定
- MCP 工具 `memo_remember` 增加 `importance` 参数：`"high"` / `"normal"`

## 五、CAS：变更感知

### 5.1 问题

Memo 已有 `is_update_of` 机制：LLM 在提取时如果识别出对话推翻了旧事实，会列出关键词 → 检索匹配旧记忆 → 标记 `is_superseded`。

但这个方法有三个致命缺陷：

| 缺陷 | 后果 |
|------|------|
| **完全依赖 LLM 在提取时「猜」旧事实关键词** | LLM 看不到记忆库实际内容，只能凭当前片段推测。猜错了关键词就匹配不上旧记忆，旧记忆继续存活 |
| **只处理显式推翻，不处理隐式更新** | 「我们不用 PostgreSQL 了，换 MySQL」→ LLM 可能识别。但「PostgreSQL 的索引策略从 B-tree 换成 GIN」→ LLM 大概率不标 `is_update_of` |
| **只在写入时做，不做定期扫描** | 写入时漏掉的冲突永远不会被发现。`conflict.py` 有框架但核心逻辑是空的 |

结果：旧方案 A 和新方案 B 同时存在于记忆库，检索时可能随机返回过时信息。

### 5.2 学术参考

- **PAMU** 的 Preference Change Perception Module：滑动窗口 + EMA 双重视角检测偏好变化
- **D-Mem** 的 Full Deliberation：绕过压缩记忆直接扫描原始对话，做 query-guided 事实提取
- **Memoria** 的 version tracking：保留完整记忆历史，在提取前检索已有记忆进行比较

### 5.3 方案设计

**核心思路**：不依赖 LLM 主动猜 `is_update_of`，而是在每次写入后，主动检索相似旧记忆，用 LLM 做对比判断。

#### 两级变更检测

**L1：写入时语义冲突检测**（实时，每次 `remember_conversation` 之后）

```
新记忆写入
  ↓
向量检索 top-5 相似旧记忆（同特征词簇、未失效）
  ↓
LLM 对比判断：新记忆是否更新/推翻/细化旧记忆？
  ↓
  ├── 是推翻 → supersede 旧记忆，记录 valid_until
  ├── 是细化 → 关联新旧记忆（新增 relation: REFINES）
  └── 无关 → 跳过
```

```python
# 新的冲突检测 prompt（替代 conflict.py 的空壳）
CONFLICT_CHECK_PROMPT = """判断以下新记忆是否更新、推翻或细化了一条旧记忆。

旧记忆：
  [{old_id}] {old_title}
  {old_summary}

新记忆：
  {new_title}
  {new_summary}

输出 JSON：
{
  "relation": "supersede" | "refine" | "unrelated",
  "reason": "简述判断原因"
}

- supersede：新记忆推翻了旧记忆的结论（如「决定用A」→「改用B」）
- refine：新记忆补充或细化了旧记忆，但未推翻（如「用PostgreSQL」→「PostgreSQL版本需≥15」）
- unrelated：两者虽相似但无更新关系"""
```

**L2：consolidation 定期扫描**（批量，`run_lifecycle()` 触发时）

```
获取同一特征词簇下 is_superseded=0 的记忆
  ↓
对每对摘要相似度 > 0.75 的记忆
  ↓
批量 LLM 判断（一次调用处理多对，节约成本）
  ↓
标记 superseded / 建立 REFINES 关系
```

#### 防止级联效应

如果 A→B→C 连续更新，不希望检索时只看到 C 却不知道有这条演化链。因此：

- `superseded_by` 字段保留反向指针：可以追溯「谁替代了我？」
- 检索时，如果返回的记忆有 `superseded_by`，可选显示「此信息已被更新」提示
- 新增 `RelationType.REFINES`：细化关系不标记失效，而是建立关联边

#### 新增 RelationType

```python
class RelationType(str, Enum):
    CO_OCCUR = "CO_OCCUR"
    CAUSAL = "CAUSAL"
    TEMPORAL = "TEMPORAL"
    DERIVED = "DERIVED"
    CONTRADICT = "CONTRADICT"
    REFINES = "REFINES"      # 新增：细化关系（非推翻）
    SUPERSEDES = "SUPERSEDES" # 新增：推翻关系
```

#### 实现要点

- `conflict.py` 重写为 `change_detector.py`，实现 `detect_change()`
- `engine.remember_conversation()` Step 6（冲突检测）改为调用 `detect_change()`
- `engine.run_lifecycle()` 的 consolidation 阶段增加变更扫描
- `memo/models` 增加 `RelationType.REFINES`、`RelationType.SUPERSEDES`
- 配置项：`MEMO_CHANGE_DETECTION_ENABLED=true`、`MEMO_CHANGE_SIMILARITY_THRESHOLD=0.75`

### 建议执行顺序

```
MVG（先治噪音，产出干净数据）
  ↓
CAS（再保证数据不冲突——干净的旧数据才值得做变更检测）
  ↓
ESA（给重要数据打标签）
  ↓
SCB（最后优化关联权重，此时数据已有质量保障和一致性保障）
```
