# Memo V0.9 Source-aware / Turn-aware 记忆重建方案

> 状态：新方案定稿草案  
> 日期：2026-07-23  
> 目标：替换“旧碎片聚合”方案，建立基于真实 Agent 会话、会话内对话颗粒和跨会话融合的长期记忆架构。  
> 关键结论：当前已有 Memo 数据库先完整备份归档，不在旧库上继续做碎片聚合。下午实现的“基于已有碎片由模型总结会话内容”的代码路径不再作为后续方向。

---

## 1. 背景

Memo 当前已有能力：

- 总览：记忆、会话、关系、待办、活跃特征词
- 记忆管理：搜索、标重要、标错误、标过期、静默、软删除
- 治理审计：去重、来源、合并、ingestion 记录
- Context Space：创建、编辑、归档、简报、记忆解绑、待办绑定
- 待办：创建、完成、重开、绑定 Space
- 人格画像：10 维人格断言管理
- 图谱：当前入口占位，后续接 Canvas 图谱
- 赫布学习：基于特征共现持续强化关系

但旧写入链路存在一个根本问题：

```text
模型中间回复 / 工具调用总结 / 阶段性总结 → 被拆成多条 memory_units
```

这会导致：

- 一个真实用户问题被拆成 N 条过程碎片。
- 会话名称被 Memo 根据碎片内容重新总结，不能代表 Agent 原始会话名。
- 图谱和 Hebbian 学到的是过程噪声，而不是真实业务关系。
- Space 边界和人格画像容易被模型话术污染。

因此，V0.9 新方向不再做“旧碎片聚合”，而是从 Agent 原始会话重新建立数据地基。

## 2. 核心判断

### 2.1 当前已有数据库处理

用户当前已有 `memo.db` 作为历史实验数据和证据资产，处理原则：

```text
完整备份 → 归档保存 → 默认不再继续写入或改造 → 新库按新架构重建
```

不建议继续在旧库内对碎片做模型聚合，因为那会把旧写入策略的问题固化成新结构。

### 2.2 会话是整合层，不是原始颗粒

会话 `source_session` 很重要，但不能成为唯一颗粒度。

正确层级是：

```text
Agent 原始会话 source_session
  ↓
会话内 turn / message / tool_call / assistant_final
  ↓
用户意图 episode / task segment
  ↓
最小长期记忆 memory_unit
  ↓
跨会话融合 graph / tags / Space / persona / todo
```

也就是说：

- 会话负责来源和边界。
- turn 负责精确证据。
- episode 负责用户意图。
- memory_unit 负责长期知识颗粒。
- graph / Space / Persona 负责跨会话融合。

### 2.3 不被会话限制死

同一会话内可能有多个项目、多个问题、多个决策。

同一项目也可能分布在多个会话里。

因此，Memo 不能只做：

```text
一个会话 = 一个记忆
```

而应该做：

```text
一个会话 = 多个 turn + 多个 episode + 多个 memory_unit
多个会话 = 可以通过标签、实体、项目、Space、语义向量和 Hebbian 关系融合
```

## 3. HanaAgent 会话标题调查结论

本机 HanaAgent 数据目录：

```text
C:\Users\Administrator\.hanako\agents\hanako\sessions\
C:\Users\Administrator\.hanako\session-manifest.db
```

关键文件：

```text
session-titles.json
session-meta.json
session-manifest.db
```

### 3.1 session-titles.json

结构：

```text
jsonl 绝对路径 → 会话标题
```

它最像 HanaAgent UI 使用的真实会话标题来源。

示例形态：

```text
C:\Users\Administrator\.hanako\agents\hanako\sessions\xxx.jsonl
→ 安装技能项目 / 检查云端记忆 / 游戏计划了解
```

### 3.2 session-manifest.db

`session_manifests` 表包含：

```text
session_id
owner_agent_id
kind
current_locator_path
current_locator_key
created_at
updated_at
provenance_json.legacyTitle
```

它是更稳定的会话身份索引，负责把 HanaAgent 内部 `session_id` 和真实 jsonl 路径绑定起来。

### 3.3 session-meta.json

不是标题来源，主要包含：

```text
memoryEnabled
experienceEnabled
workspaceFolders
authorizedFolders
permissionMode
accessMode
thinkingLevel
promptSnapshot
memoryReflectionSnapshot
toolNames
```

### 3.4 JSONL 内部

抽样 JSONL 前 50 条，主要类型为：

```text
type=session
type=model_change
type=thinking_level_change
type=message
```

未发现 title/name 字段。

### 3.5 标题覆盖率

当前调查样本：

```text
manifest chat sessions: 41
session-titles entries: 49
covered by session-titles or legacyTitle: 12
missing title: 29
```

说明很多会话有真实 session/path，但没有已生成或已保存的标题。

### 3.6 HanaAgent 标题获取优先级

```text
1. session-titles.json[jsonl_abs_path]
2. session-manifest.db.provenance_json.legacyTitle
3. 无标题：source_session.title = null，title_source = missing
4. 可生成 fallback_display_title，但必须标记 title_source = generated_fallback
```

重要原则：

```text
真实 Agent 会话名 ≠ Memo 根据内容重新总结的标题
```

Memo 生成的标题只能作为展示 fallback，不能写入 `source_session.title` 冒充 Agent 原始标题。

## 4. 主数据模型

### 4.1 source_sessions：会话身份层

保存 Agent 原始会话身份。

建议字段：

```text
id
source_agent              hanaagent / workbuddy / qoder / codex / claude / cursor / generic
agent_session_id          Agent 内部 session id
source_path               原始文件 / DB / export 路径
source_hash               文件或会话内容 hash
original_title            Agent 原始标题，可为空
title_source              agent_title / legacy_title / missing / generated_fallback
display_title             UI 展示标题，可来自 fallback
created_at
updated_at
imported_at
metadata_json
```

### 4.2 source_turns：会话内具体对话颗粒

这是新方案的关键。会话不是原始颗粒，turn 才是证据颗粒。

建议字段：

```text
id
source_session_id
agent_turn_id
parent_turn_id
role                       user / assistant / tool / system
content
content_hash
timestamp
turn_index
is_final_answer
is_tool_call
is_tool_result
tool_name
metadata_json
```

### 4.3 episodes：会话内任务/问题边界

一个 episode 表示一次用户问题、任务、决策或阶段性推进。

```text
id
source_session_id
title
user_intent
start_turn_id
end_turn_id
start_turn_index
end_turn_index
status                    candidate / active / ignored
confidence
created_at
updated_at
metadata_json
```

### 4.4 episode_turns：episode 与 turn 的证据映射

```text
episode_id
turn_id
role_in_episode            question / evidence / tool_trace / final_answer / correction
weight
```

### 4.5 memory_units：最小长期记忆颗粒

`memory_units` 仍然保留，但写入来源变成 episode/turn-aware。

新增或扩展字段：

```text
source_session_id
episode_id
source_turn_start_id
source_turn_end_id
memory_granularity         atomic / episode_summary / decision / preference / fact / todo_hint
speaker_scope              user_claim / assistant_result / tool_observation / decision / mixed
source_confidence
is_canonical
```

### 4.6 memory_turn_sources：记忆与具体 turn 的关联

为了支持跨会话融合和可追溯，记忆不只关联会话，也要关联具体对话数据。

```text
memory_id
turn_id
evidence_role              primary / support / contradiction / tool_evidence / final_answer
weight
```

## 5. AgentAdapter 设计

统一接口：

```python
class AgentAdapter:
    def detect(self) -> list[AgentSource]
    def list_sessions(self, source) -> list[RawSessionRef]
    def load_session(self, ref) -> RawSession
    def normalize_turns(self, raw_session) -> list[SourceTurn]
    def get_original_title(self, raw_session) -> TitleInfo
    def get_metadata(self, raw_session) -> dict
```

### 5.1 首批适配目标

| Agent | 优先级 | 标题/会话来源策略 |
|---|---:|---|
| HanaAgent | P0 | `session-manifest.db` 建身份索引，`session-titles.json` 取真实标题，JSONL 取 turn |
| WorkBuddy | P0 | 优先读取 `~/.workbuddy/workbuddy.db.sessions.custom_title/title`，再读取本地会话/导出 |
| Qoder / QoderWork | P0 | 自动探测 `~/.qoder` / `~/.qoderwork`，优先官方会话标题/项目标题，缺失则标 missing |
| Codex / Codex CLI | P1 | 支持用户指定 CLI transcript/log 目录，优先官方 session id/title，先走 generic transcript adapter |
| Claude Desktop | P1 | 先支持用户导出文件，若有 conversation title 则保留 |
| Cursor | P1 | 先支持 workspace chat export / 用户导出文件 |
| ChatGPT / OpenAI Export | P2 | 支持官方 data export zip / conversations.json |
| Gemini / 通用网页 Agent | P2 | 支持 Markdown / HTML / JSON 导出 |
| 通用 MCP Agent | P0 | 只要能导出 User/Assistant/Tool 序列，即可接入 generic adapter |

## 6. 新导入链路

### 6.1 clean import

```text
AgentAdapter.detect/list/load
  ↓
读取真实 session id / title / path / created_at / updated_at
  ↓
写入 source_sessions
  ↓
解析并写入 source_turns
  ↓
按用户问题、任务边界切 episodes
  ↓
从 episode 中提取最小 memory_units
  ↓
建立 memory_turn_sources / episode_turns
  ↓
提取 tags / vectors
  ↓
更新 Hebbian / graph / Space / Persona / Todo candidates
```

### 6.2 旧库处理

当前已有数据库：

```text
E:\memo\memo\data\memo.db
```

处理原则：

```text
1. 完整备份
2. 归档为 old_memo_before_source_aware_YYYYMMDD.db
3. 新建干净库
4. 不从旧 memory_units 直接聚合生成新记忆
5. 只在需要时从旧库查询历史，或迁移能明确回到 source_session/turn 的数据
```

## 7. 记忆提取粒度

### 7.1 不要一个会话一条记忆

因为一个会话里可能包含多个主题。

### 7.2 不要一个模型回复一条记忆

因为模型回复可能是过程性噪声。

### 7.3 推荐粒度

```text
一个用户问题 / 一个任务阶段 / 一个明确决策 / 一个事实更新 → 1 到 N 条 atomic memory_units
```

示例：

```text
用户问：安装包是不是把数据库打进去了？

可提取：
1. 发现 desktop package 曾误包含 memo/data/memo.db 和 backups。
2. 原因是 package.json files 包含 memo/**/*，旧排除规则未排 memo/data。
3. 修复方式是添加 !**/data/**/*、!**/*.db 等强排除。
4. 后续发布流程必须扫描 win-unpacked。
```

它们属于同一个 episode，但可以是多条最小长期记忆。

## 8. Hebbian / 图谱重构

旧 Hebbian 逻辑的问题不是“持续学习”错了，而是输入太脏。

新规则：

```text
同一 episode 内的高质量 memory/tag 共现：强关系
同一 source_session 内普通共现：中等关系
跨多个真实 source_session 反复出现：长期稳定关系
用户显式偏好 / 决策：高权重关系
工具日志 / 模型中间话术：不参与或低权重参与
missing title / generated fallback 不作为真实会话关系证据
```

关系维度建议：

```text
co_episode
co_session
cross_session_repeat
supports_decision
contradicts
supersedes
belongs_to_space
mentions_entity
```

图谱主节点：

```text
memory_unit
feature_tag
entity
source_session
episode
space
persona_assertion
todo
```

图谱主视图默认隐藏工具过程和 fragment，只展示高质量长期记忆与项目/Space/实体关系。

## 9. Space / Persona / Todo 的新关联

### 9.1 Space

Space 不直接受会话限制。

绑定策略：

```text
memory_unit → Space
episode → Space
source_session → Space 仅作为辅助归属
```

一个会话可包含多个 Space；一个 Space 可跨多个会话。

### 9.2 Persona

只从高置信度、长期价值高的 memory_units 中提取人格断言。

优先来源：

```text
user_claim
decision
preference
repeated cross-session behavior
```

禁止来源：

```text
tool log
assistant process narration
generated fallback title
```

### 9.3 Todo

Todo 候选从 episode 和 memory_unit 中提取，但必须保留 turn 证据。

```text
todo → memory_unit
todo → episode
todo → source_turn evidence
```

## 14. 最终原则

1. 旧数据库先备份归档，不继续在旧碎片上修花园。
2. 新库从 Agent 原始会话重建，不从模型总结的旧碎片重建。
3. 会话是整合层，不是原始颗粒。
4. 具体 turn 是证据颗粒。
5. episode 是用户意图颗粒。
6. memory_unit 是长期知识颗粒。
7. 跨会话融合依靠 tag/entity/vector/Hebbian/Space，而不是把所有东西压成会话摘要。
8. 真实 Agent 标题和 Memo fallback 标题必须区分。
9. 工具过程和模型中间话术不参与主图谱强化。
10. Hebbian 学习继续存在，但必须 source-aware / episode-aware / turn-aware。
