# Memo V0.9 Episode Memory 与 Agent 历史迁移方案

> 状态：设计定稿草案  
> 目标版本：V0.9.x  
> 适用范围：新用户、已有 Memo 用户、已有 HanaAgent / WorkBuddy / Qoder / Codex / Claude / Cursor 等 Agent 历史会话用户  
> 核心原则：最小记忆可以来自单轮对话，但必须保留原始 Agent 会话归属；不要把聊天记录本身当长期记忆，而是从带来源归属的会话中提取长期认知。

---

## 1. 背景与问题

当前 Memo 已具备记忆写入、检索、Space、人格画像、治理审计、待办、图谱入口等能力，但历史写入粒度偏细：

- 一个用户问题可能触发多次模型中间回复、工具调用、阶段总结和最终回复。
- 这些中间过程被拆成多条 `memory_units` 后，会污染长期记忆层。
- 图谱关系、Space 边界、活跃特征词和人格画像容易被模型过程词、工具过程词、重复阶段总结拉歪。

典型问题：

```text
用户在一个会话里问 3 个问题；
模型为了回答每个问题调用 N 次工具；
旧逻辑可能写入 N 条阶段性模型总结；
最终记忆库里出现大量“我正在检查”“准备测试”“已经开始打包”这类过程碎片。
```

这对长期记忆系统不理想。Memo 的目标不是聊天记录仓库，而是长期上下文中枢。

---

## 2. 总体判断

V0.9 应将长期记忆主对象从：

```text
message-level memory / 模型过程碎片
```

升级为：

```text
episode-level canonical memory / 用户意图级规范记忆
```

一句话原则：

```text
长期记忆不记录聊天过程，记录完成后的认知增量。
```

更工程化地说：

```text
raw session 是日志；
source_session 是原始 Agent 会话身份与证据链；
episode 是 source_session 内的用户意图 / 单轮或多轮任务边界；
memory_unit 是 episode 下最小颗粒度的长期事实 / 决策 / 偏好 / 事件；
canonical memory 是对同一 source_session / episode 下碎片的规范化沉淀。
```

### 2.1 2026-07-23 方向修订：重新开始时必须保留真实会话归属

经过对现有实现的复盘，V0.9 的关键不只是“把记忆变大”或“重新总结会话标题”，而是要保证每条最小颗粒度记忆都有稳定、可追溯的原始会话归属。

用户可以选择清空旧历史数据、归档旧库并重新跑导入。但重新开始后的数据模型必须满足：

```text
Agent 原始会话（HanaAgent / WorkBuddy / Qoder / Codex ...）
  ↓ source_session：保存真实会话 id、真实会话标题、来源路径、时间戳、Agent 名称
Episode / Turn：source_session 内的单个用户问题、任务或阶段边界
  ↓
memory_unit：最小颗粒度长期记忆，可按单轮对话提取
```

也就是说，**记忆可以是最小颗粒度，但不能失去会话归属**。

必须避免再次出现：

```text
所有实时同步记忆都写入“自动同步会话”
→ memory_units 只剩碎片
→ 无法按 HanaAgent / WorkBuddy 真实会话名溯源
→ 会话层整理只能重新生成展示名，而不是恢复真实会话名
```

后续实现必须优先保证：

1. `source_sessions.title` 使用 Agent 中真实会话名，不用 LLM 重新总结替代。
2. `source_sessions.external_session_id` 保存 Agent 原始 session id / thread id。
3. `source_sessions.source_path` 保存原始 JSONL / DB / 导出文件路径或稳定来源引用。
4. 每条 `memory_units` 必须能通过 `source_session_memories` 或 `episode_id` 回到原始 source_session。
5. 即使 memory_unit 是“单轮对话级别”，也必须知道它属于哪个完整会话。
6. 后续 Hebbian 关系、图谱、Space、Persona、Todo 候选都要基于 source-aware / episode-aware 数据逻辑重新计算，而不是继续基于无来源碎片直接强化。

这意味着 V0.9 不只是一个迁移工具，而是一次底层数据治理与图谱计算逻辑的完整改造。

## 3. 核心概念

### 3.1 Source Session

原始会话来源层。保存来自不同 Agent 的会话、消息、工具调用、最终回复、时间戳和来源路径。

作用：

- 可追溯
- 可重新导入
- 可调试
- 可证明某条记忆来自哪里

### 3.2 Episode

一次用户意图、问题、任务、决策或阶段性推进。

默认切分规则：

- 一个新的用户问题通常开启一个新 episode。
- 用户明显切换任务时开启新 episode。
- 一次重大事故、发布决策、架构决策可独立成 episode。
- 模型中间总结、工具调用、阶段性回复归入当前 episode，不直接写长期记忆。

### 3.3 Canonical Memory

从 episode 中提炼出的长期记忆。它是 Memo 后续检索、图谱、Space、人格画像、待办绑定的主对象。

建议结构：

```text
标题：
用户意图：
关键事实：
处理过程：
最终结论：
决策 / 结果：
后续影响：
来源：source_session + turn range + source_memory_ids
```

### 3.4 Fragment Memory

历史遗留的碎片记忆或过程性记忆。它们不删除，但默认降级为证据层：

- `superseded`
- `muted`
- linked to canonical episode

## 4. 新用户策略

新用户从第一天开始使用 episode-level canonical memory。

### 4.1 新写入流程

```text
用户发起问题
  ↓
创建或打开 current_episode
  ↓
模型工具调用 / 阶段总结 / 中间回复
  ↓
写入 episode buffer，不直接写 memory_units
  ↓
最终回复完成
  ↓
生成或更新 canonical memory
  ↓
提取 tags / vectors / relations / Space / persona / todo candidates
```

### 4.2 新用户体验

用户不需要理解 episode、source_session、canonical memory。

产品呈现：

```text
安装 Memo
配置 Key，可跳过
接入 Agent
正常聊天
Memo 自动整理长期记忆
```

后台自动完成：

```text
raw conversation → episode → canonical memory → graph/space/persona/todo
```

## 5. 老用户策略

老用户分两类：

1. 已有 Memo 数据库。
2. 没有 Memo，但已有 HanaAgent / WorkBuddy / Qoder / Codex 等历史会话。

### 5.1 已有 Memo 数据库用户

推荐路线：

```text
旧库归档 + 新库重建 + 精选历史重导入
```

不要直接删除旧库。

流程：

```text
1. 停止 Memo 服务
2. 备份当前 memo.db
3. 将旧库归档为 old_memo_before_episode_YYYYMMDD.db
4. 创建全新 memo.db
5. 启用 episode-level 新写入逻辑
6. 从旧库 / source_sessions / Agent 原始会话中提取高价值 episode
7. 导入为 canonical memories
8. 重建 tags、relations、vectors、spaces、persona
9. 旧库作为历史档案保留，不参与日常检索
```

可选保守路线：

```text
原库内自动重铸：生成 canonical memories，旧碎片标记 superseded/muted，并建立 memory_links。
```

但 V0.9 alpha 阶段，优先推荐“归档旧库，新库重建”，更干净，更适合验证新模型。

### 5.2 已有 Agent 历史但无 Memo 用户

推荐产品入口：

```text
历史记忆迁移
```

不要叫“导入聊天记录”。

页面文案建议：

```text
Memo 会扫描你过去在 Agent 中的重要对话，提取项目进展、偏好、决策和待办线索。
模型中间回复、工具日志和临时闲聊会自动跳过。
导入前你可以预览和取消。
```

迁移流程：

```text
选择来源 Agent
  ↓
扫描历史会话
  ↓
切分 episode
  ↓
评分 long_term_value_score
  ↓
生成候选 canonical memories
  ↓
用户预览 / 勾选 / 取消
  ↓
导入
  ↓
生成 Space / persona / todo candidates
```

## 6. 历史导入模式

提供三种模式：

### 6.1 推荐模式（默认）

只导入高价值 episode：

- 项目进展
- 明确决策
- 用户偏好
- 长期事实
- 重要账号/环境/路径/部署信息
- 待办线索
- 事故与修复结论

自动跳过：

- 模型中间总结
- 工具调用过程
- 临时寒暄
- 重复阶段性回复
- 无长期价值的报错尝试

### 6.2 全面模式

尽量导入所有有意义 episode，适合重度用户或研究型用户。

仍然跳过纯工具日志和模型自述。

### 6.3 手动模式

先生成候选列表，由用户按 Agent、时间、项目、Space、价值评分勾选。

## 7. Agent 兼容策略

核心原则：Agent 兼容层只负责读取、标准化和追溯，不直接决定长期记忆内容。

统一抽象：

```text
AgentAdapter
  - detect(): 是否存在该 Agent 数据
  - list_sessions(): 列出会话
  - load_session(): 读取会话
  - normalize_turns(): 转成 Memo 标准 turn 格式
  - extract_tool_traces(): 标准化工具调用
  - source_metadata(): 来源、路径、版本、账号标识
```

标准 turn 格式：

```json
{
  "agent": "hanaagent",
  "session_id": "...",
  "turn_id": "...",
  "role": "user | assistant | tool | system",
  "content": "...",
  "timestamp": "...",
  "tool_name": "optional",
  "is_final": true,
  "metadata": {}
}
```

### 7.1 首批 Agent 适配目标

| Agent | 优先级 | 数据来源策略 | 说明 |
|---|---:|---|---|
| HanaAgent | P0 | session JSONL / SessionFile / 本地会话目录 | Memo 自身核心使用场景，必须优先支持 |
| WorkBuddy | P0 | 本地会话目录 / MCP export / JSONL | 已有使用基础，必须优先支持 |
| Qoder / QoderWork | P0 | 本地配置目录 / 会话导出 / MCP 可读源 | 面向开发场景，优先支持 |
| Codex / Codex CLI | P1 | CLI session logs / transcript export / workspace logs | 以通用 JSONL/Markdown transcript 适配为主 |
| Claude Desktop | P1 | 用户导出文本 / 项目会话导出 / MCP 侧导出 | 本地原始库结构不稳定，先支持导入文件 |
| Cursor | P1 | workspace chat export / MCP config / 用户导出 | 先支持手动选择目录或导出文件 |
| ChatGPT / OpenAI 导出 | P2 | 官方 data export zip / conversations.json | 可作为长线兼容 |
| Gemini / 通用网页 Agent | P2 | Markdown / HTML / JSON 导出 | 走通用导入器 |
| 通用 MCP Agent | P0 | `memo_export` / JSONL / Markdown transcript | 只要能导出 User/Assistant/Tool 序列即可 |

### 7.2 兼容原则

- 不假设所有 Agent 都有稳定本地数据库。
- 优先支持“用户选择文件/文件夹导入”。
- 对已知 Agent 提供自动探测。
- 所有来源都转成统一 source_session + turns。
- 不把 Agent 内部工具调用直接写长期记忆。
- 保留 source path / hash，避免重复导入。

## 8. 质量评分规则

每个 episode 生成 `long_term_value_score`。

加分项：

- 包含用户偏好
- 包含项目进展
- 包含明确决策
- 包含技术修复结论
- 包含发布/部署/账号/路径/环境信息
- 包含风险或事故复盘
- 包含待办或计划
- 多次会话反复出现

扣分项：

- 只有模型过程
- 只有工具日志
- 只有临时寒暄
- 与已有 canonical memory 高度重复
- 没有长期上下文价值
- 只是一次失败尝试且没有结论

默认阈值建议：

```text
score >= 0.70：推荐导入
0.45 <= score < 0.70：进入人工确认
score < 0.45：默认跳过
```

## 9. Dashboard 调整

### 9.1 总览

新增口径：

```text
记忆库总量：全部 memory_units
有效长期记忆：active canonical memories
历史碎片证据：superseded / muted fragments
待治理保留：未归类或低置信度片段
会话来源：source_sessions
关系数量：基于 active canonical memories 的关系
活跃特征词：优先来自 canonical memories
```

### 9.2 记忆管理

默认搜索：

```text
active canonical memories
```

高级筛选：

- 全部
- Canonical
- Fragment
- Superseded
- Muted
- Deleted
- By Agent
- By Space
- By Import Run

### 9.3 治理审计

新增：

- Import Runs
- Episode Split Reports
- Canonicalization Reports
- Skipped Low-value Episodes
- Fragment → Canonical links
- Rebuild Graph Reports

### 9.4 Space

Space 默认绑定 canonical memories。

旧 fragment 通过 canonical memory 间接归属，不直接撑大 Space 边界。

### 9.5 图谱

Canvas 图谱主对象：

- canonical episode memory
- feature tag
- space
- persona assertion
- todo

旧 fragment 只在点击 canonical memory 后作为证据链展开。

## 10. 赫布学习与持续优化

赫布学习继续保留，但改为基于高质量输入。

旧问题不是赫布学习无意义，而是输入粒度太碎。

新规则：

```text
canonical episode：正常 Hebbian 强化
user decision / preference：高权重强化
tool/process fragment：不强化或极低权重
superseded / muted fragment：不参与 active graph 强化
```

持续优化流程：

```text
ingestion
  ↓
episode splitting
  ↓
canonicalization
  ↓
quality gate
  ↓
tag/vector extraction
  ↓
graph update
  ↓
space/persona/todo candidate refresh
  ↓
governance audit
```

定时任务不再只是追加记忆和维护权重，而是持续整理：

- 新 episode 合并
- 低质量片段静默
- 重复 canonical memory 候选
- 过期状态检测
- Space 简报刷新
- 人格断言增量审计

## 11. 数据库与迁移建议

### 11.1 最小新增字段 / 表

建议新增：

```text
episodes
canonicalization_runs
episode_sources
import_runs
agent_sources
```

如果尽量少加表，可以先复用：

```text
memory_units
memory_links
source_sessions
source_session_memories
```

但为了长期清晰，推荐新增 episode 层。

### 11.2 推荐表结构草案

```sql
CREATE TABLE episodes (
  id TEXT PRIMARY KEY,
  source_session_id TEXT,
  agent_name TEXT,
  title TEXT,
  user_intent TEXT,
  start_turn_id TEXT,
  end_turn_id TEXT,
  status TEXT DEFAULT 'active',
  confidence REAL DEFAULT 0,
  created_at TEXT DEFAULT CURRENT_TIMESTAMP,
  updated_at TEXT DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE episode_sources (
  episode_id TEXT NOT NULL,
  source_type TEXT NOT NULL,
  source_id TEXT NOT NULL,
  role TEXT,
  weight REAL DEFAULT 1.0,
  PRIMARY KEY (episode_id, source_type, source_id)
);

CREATE TABLE import_runs (
  id TEXT PRIMARY KEY,
  source_agent TEXT,
  source_path TEXT,
  mode TEXT,
  status TEXT,
  scanned_sessions INTEGER DEFAULT 0,
  candidate_episodes INTEGER DEFAULT 0,
  imported_memories INTEGER DEFAULT 0,
  skipped_items INTEGER DEFAULT 0,
  report_json TEXT,
  created_at TEXT DEFAULT CURRENT_TIMESTAMP,
  completed_at TEXT
);

CREATE TABLE canonicalization_runs (
  id TEXT PRIMARY KEY,
  mode TEXT,
  status TEXT,
  input_memory_count INTEGER DEFAULT 0,
  output_memory_count INTEGER DEFAULT 0,
  superseded_count INTEGER DEFAULT 0,
  muted_count INTEGER DEFAULT 0,
  report_json TEXT,
  created_at TEXT DEFAULT CURRENT_TIMESTAMP,
  completed_at TEXT
);
```


## 14. 安全与回滚

强制规则：

- apply 前必须备份数据库。
- 不删除旧库。
- 不覆盖 `.env`。
- 不导入 API key、密码、token 等敏感字段为普通记忆；敏感信息需要单独标记或跳过。
- 每次 import / canonicalization 都生成 report。
- 所有导入项保留 source hash，避免重复导入。

回滚方式：

```text
停止服务 → 替换 memo.db 为备份 → 重启服务
```

## 18. 重新开始方案：Source-aware 最小记忆架构

### 18.1 新判断

如果旧历史数据已经因为“自动同步会话”、缺失真实 Agent session id、缺失真实会话标题而难以可靠回填，可以接受：

```text
完整备份旧库
→ 归档旧历史数据
→ 新建干净库
→ 从 Agent 原始会话重新跑导入
```

但这不是简单清空重来，而是按新的 source-aware 架构重建。

### 18.2 新写入链路

新链路必须是：

```text
AgentAdapter.detect/list/load
  ↓
读取 Agent 原始会话：真实 session id、真实 title、source path、created_at、updated_at
  ↓
写入 source_sessions
  ↓
按用户问题 / 任务边界切 episode
  ↓
从 episode 中提取最小 memory_units
  ↓
建立：source_session → episode → memory_unit 的映射
  ↓
基于 source-aware memory_units 重建 tags / relations / vectors / spaces / persona / todo candidates
```

其中：

- `source_session` 是会话身份层。
- `episode` 是会话内部的任务/问题边界。
- `memory_unit` 是最小长期记忆颗粒度。
- `canonical memory` 是可选的会话级/episode级沉淀，不应该替代 source_session 身份。

### 18.3 Agent 真实会话名要求

不同 Agent 的真实标题来源不同：

| Agent | 标题来源原则 |
|---|---|
| WorkBuddy | 优先读取 `~/.workbuddy/workbuddy.db.sessions.custom_title/title` |
| HanaAgent | 优先查 HanaAgent 会话索引/元数据；若 JSONL 无 title，则需要复刻 HanaAgent UI 的标题来源逻辑或读取其真实索引，不得用 Memo 自己的 LLM 标题冒充真实标题 |
| Qoder / Codex / Claude / Cursor | 优先使用各自导出的 session title / conversation title / project title；没有真实标题时明确标记为“无原始标题”，不能伪装成真实标题 |

重要原则：

```text
真实 Agent 会话名 ≠ Memo 根据记忆内容重新总结的标题
```

重新总结标题只能作为 fallback display_title，不能作为 source_session.title 的事实来源。

### 18.4 旧库处理原则

如果选择重新开始：

1. 必须先完整备份旧 `memo.db`。
2. 旧库归档，不直接删除。
3. 新库只接受符合 source-aware 规则的新导入数据。
4. 如果从旧库迁移，只迁移能明确回到真实 source_session 的数据。
5. 无法确定真实来源的旧 memory_units 默认不进入新库，可留在旧库查询。

### 18.5 图谱与赫布学习重构

新数据逻辑上线后，不能继续用旧逻辑直接基于所有 memory_units 强化图谱。

新的 Hebbian / 图谱计算应至少区分：

```text
同一 source_session 内共现
同一 episode 内共现
跨 source_session 重复出现
用户显式偏好 / 决策
工具过程 / 临时日志
canonical memory 与 fragment memory
```

建议规则：

- episode 内共现权重高于 source_session 内普通共现。
- 跨多个真实会话反复出现的标签/关系，才增强为长期稳定关系。
- 工具过程、模型中间话术、失败尝试不参与或低权重参与图谱。
- canonical memory 参与主图；fragment memory 默认作为证据链，不直接撑大图谱。
- Space / Persona / Todo 候选优先基于 high-value episode / canonical memory，而不是全部碎片。

### 18.6 下一步开发顺序

新的优先级应调整为：

```text
P0. 找到并确认各 Agent 的真实会话标题来源
P0. 修正 watcher：实时同步时按原始 Agent session 建立 source_session，而不是统一写入“自动同步会话”
P0. 实现 source-aware clean import：从 Agent 原始会话重建 source_sessions / episodes / memory_units
P1. 设计旧库归档与新库初始化流程
P1. 基于新数据重新生成 vector / tags / feature_relations
P1. Dashboard 展示真实 Agent 会话名、episode、最小记忆和证据链
P2. 用户确认后再启用 canonicalization / fragment mute / graph rebuild
```

### 18.7 晚间交接结论

当前最终共识：

```text
可以重新开始；
可以把记忆拆到单轮/最小对话颗粒度；
但每条记忆必须保留真实 Agent 会话归属；
会话名称必须来自 Agent 原始会话名；
图谱、赫布、Space、Persona 都要按 source-aware 新逻辑重建。
```

这个结论优先级高于前文中“仅做粘贴导入预览”或“用记忆内容推导会话展示名”的临时方案。
