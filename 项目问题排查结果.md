# 项目问题排查结果

检查路径：`D:\个人\Hanako项目文件\Memo_V0.1.0`

检查方式：只读检查项目结构、关键源码、配置文件、文档、Git 状态与数据文件状态。除本报告文件外，未修改项目其他内容。

---

## 一句话结论

这个项目已经不是简单 Demo，而是一个功能较完整的 **本地 SQLite 记忆系统 + MCP Server + 看板 + 人格画像 + 待办管理**。

但目前存在几个会影响实际部署稳定性的关键问题：**数据库路径不一致、Bridge watcher 有明显运行 bug、依赖声明不完整、文档版本混乱、`.env` 中存在真实 API Key**。

综合评价：**原型完成度 70% 左右，工程化/可交付完成度 50%～60%。**

---

## 1. 项目结构与定位

项目主要模块：

| 模块 | 作用 |
|---|---|
| `memo/core/engine.py` | 统一记忆引擎入口 |
| `memo/store/` | SQLite 数据库、记忆、图谱、向量存储 |
| `memo/extraction/` | 记忆提取、门控、冲突检测、去重 |
| `memo/retrieval/` | RRF 融合、图扩散检索 |
| `memo/persona/` | 人格画像提取、路由、偏离检测 |
| `memo/todo/` | 待办管理 |
| `memo/mcp/server.py` | MCP Server，暴露记忆工具 |
| `scripts/memo_dashboard.py` | 本地 Web 看板 |
| `scripts/memo_watcher.py` | 自动同步守护进程 |
| `data/memo.db` | 当前存在的实际数据库文件 |

设计方向是对的：**本地 SQLite + 向量检索 + FTS5 + 图扩散 + MCP 接入 + 人格层**，这个产品结构是成立的。

## 2. 主要优点

### 2.1 架构思路比较完整

它不是单纯存聊天记录，而是分了：

- 会话 `sessions`
- 记忆单元 `memory_units`
- 特征词 `feature_tags`
- 特征关系 `feature_relations`
- 人格断言 `persona_assertions`
- 待办 `todos`

这说明项目已经有比较清晰的“记忆图谱”设计，而不是简单 CRUD。

### 2.2 检索链路有产品差异化

当前实现了三通道检索：

1. 向量语义检索
2. SQLite FTS5 全文检索
3. 图扩散激活检索

再用 RRF 融合。这套思路比较适合做“跨 Agent 记忆”，不是纯 RAG。

### 2.3 数据安全意识有所加强

`database.py` 里对 WAL 的处理是 checkpoint，不是删除 WAL 文件。这个方向是正确的：

```python
PRAGMA wal_checkpoint(TRUNCATE)
```

而且注释里明确写了“绝不删除数据”。这一点很重要。

### 2.4 MCP 工具已经比较丰富

代码里实际 MCP 工具已经不止 README 说的 11 个，而是接近 18 个，包括：

- 记忆写入/检索
- 人格画像
- 导出 Bridge
- 导入历史会话
- 待办增删查改
- 风险检测

说明功能边界已经扩展到了“记忆 + 人格 + 待办”的个人智能层。

## 3. 高优先级问题

### P0：`.env` 里存在真实 API Key

项目根目录下有 `.env`，里面不是示例值，而是真实 key。

虽然 `.gitignore` 已经忽略 `.env`，但本地压缩、分享、发包时仍然可能误带出去。

建议优先处理：

- 不要把 `.env` 放进交付包
- 发布包只保留 `.env.example`
- 如这个 key 已经暴露给别人，建议直接废弃/轮换

这是当前最紧急的安全问题。

---

### P0：数据库路径严重不一致

实际存在的数据库在：

```text
D:\个人\Hanako项目文件\Memo_V0.1.0\data\memo.db
```

大小约 **36.6MB**。

但代码默认路径是：

```text
memo/data/memo.db
```

README、SKILL.md、mcp_config.example.json 也多处写的是：

```text
memo/data/memo.db
```

问题是：当前项目里 `memo/data/` 这个目录不存在。

这意味着如果直接按 README 或默认配置启动，可能会新建一个空库，而不是读取现有 `data/memo.db`。

这类问题非常危险，因为表面看服务能启动，但用户会以为“记忆丢了”。

建议统一成一个路径，优先确认到底要用：

```text
data/memo.db
```

还是：

```text
memo/data/memo.db
```

在未确认前，千万不要执行初始化或迁移操作。

---

### P0：`memo_watcher.py` 的 Bridge 导入逻辑有明显 bug

在 `_process_inbox()` 中，有这一句：

```python
session_id = engine.start_session(f"Bridge-{agent_name}", agent_id=agent_name).id
```

但此时 `agent_name` 还没有定义。

它是在后面读取文件内容时才赋值：

```python
agent_name = record.get("agent", "unknown")
```

所以 Bridge inbox 一旦有文件要处理，这里会直接报错。

影响：

- `memo_export` 可以导出文件
- 但 watcher 自动导入可能失败
- 跨 Agent Bridge 链路不完整

这是功能级断点。

---

### P1：依赖声明不完整

`memo/mcp/server.py` 使用了：

```python
from mcp.server import Server
```

但 `requirements.txt` 里没有 `mcp`。

`pyproject.toml` 里虽然有：

```toml
[project.optional-dependencies]
mcp = ["mcp>=1.0"]
```

但用户按 README 执行：

```bash
pip install -r requirements.txt
```

不会安装 MCP 依赖。

结果就是：按说明安装后，MCP Server 可能直接启动失败。

建议把 `mcp>=1.0` 加进 `requirements.txt`，或者 README 明确写：

```bash
pip install -e ".[mcp]"
```

---

### P1：环境变量命名不一致

`.env.example` 里是：

```env
LLM_BASE_URL=https://api.deepseek.com/v1
```

`config.py` 读取的也是：

```python
LLM_BASE_URL
```

但实际 `.env` 里出现的是：

```env
OPENAI_BASE_URL=...
```

这意味着真实运行时可能不会读取到预期的 base url，而是回退到默认值。

如果默认值刚好一样，看起来没问题；但一旦换供应商，这里会出隐性 bug。

## 4. 中优先级问题

### P1：文档版本混乱

README 标题写：

```text
V0.7.0
```

但安装部分还写：

```text
Memo_V0.6.0.zip
```

项目目录名是：

```text
Memo_V0.1.0
```

`pyproject.toml` 版本是：

```toml
version = "0.1.0"
```

CHANGELOG 只到 v0.2.0，但代码和 docs 已经有 v0.6 / v0.7。

这会影响交付判断：别人不知道当前到底是哪一版，也不知道哪些功能是稳定的。

---

### P1：测试文件会写真实数据库

`tests/test_integration_phase1.py` 和 `tests/test_mcp_smoke.py` 都不是纯只读测试，会：

- 初始化数据库
- 创建 session
- 写入 memory
- 执行生命周期
- 写入待办/记忆

所以这些测试不能直接在生产数据库上跑。

建议后续加测试专用 DB，例如：

```env
MEMO_DB_PATH=data/test_memo.db
```

或者测试里强制用临时目录。

---

### P2：看板是单文件硬编码，后期维护成本高

`scripts/memo_dashboard.py` 把 HTML、CSS、JS、API handler 全放在一个 Python 文件里，已经超过 1000 行。

短期方便，但后期维护会越来越难。

如果继续产品化，建议拆成：

```text
dashboard/
  index.html
  app.js
  style.css
scripts/memo_dashboard.py
```

---

### P2：文档有编码异常

部分文档如 `docs/testing-guide.md`、`docs/setup-guide.md`、`docs/deploy-guide.md` 出现乱码，例如：

```text
涓嶄緷璧?MCP銆...
```

说明历史保存或编码转换出过问题。

这会影响对外交付观感。

## 5. 当前 Git 状态

只读执行 `git status --short` 后，发现当前有两个文件已被修改但未提交：

```text
 M memo/persona/extractor.py
 M memo/utils/llm.py
```

diff 显示是把 JSON 正则从贪婪匹配改成了非贪婪匹配：

```python
r"\{.*\}"  ->  r"\{.*?\}"
r"\[.*\]"  ->  r"\[.*?\]"
```

这个修改方向看起来是为了解决 LLM 输出里多个 JSON 块时误匹配的问题。

但这种非贪婪匹配也可能在 JSON 内部嵌套对象/数组时截断，后续最好换成更稳的 JSON 提取器，而不是继续靠正则。

| 产品方向 | 8/10 | 记忆 + 人格 + MCP + 待办，方向成立 |

**这是一个有产品潜力的本地记忆系统原型，但还不适合直接打包交付。下一步不该急着加功能，而是先做“路径统一 + 数据安全 + 启动链路验证”。**
